# Epic 3 Retrospective — Task Marketplace & Lifecycle

**Date:** 2026-04-05
**Epic:** 3 — Task Marketplace & Lifecycle
**Stories:** 3.1 · 3.2 · 3.3 · 3.4 · 3.5
**Status at retro:** All 5 stories done

---

## What Went Well

### Schema-First Foundation Paid Off
The `tasks` table, `taskStatusEnum`, `clientTypeEnum`, and Zod schemas were all in place before story 3.1 started. The epic never had to debate data shape — development moved straight to building UI and hardening business logic.

### TRPCError Migration Was Systematic
Story 3.1's explicit task of replacing every `new Error(...)` with `TRPCError` was well-scoped and executed cleanly. This prevented a class of 500-for-all-errors bugs that would have been painful to debug in demo.

### Agent/Human Separation is Clean
The `client_type` enum + separate `client_nullifier` / `client_agent_wallet` columns give the marketplace a clean dual-client model. The `validate` guard (`FORBIDDEN` for agent tasks via tRPC) correctly forces agent validation through the MCP path.

### Progressive UI Quality
Stories 3.2–3.5 built from basic CRUD to a polished task list with agent/human visual distinction and "delight" completion animations. The incremental approach kept each story shippable.

## What Was Difficult

### Agent Task Validation Bug (Story 3.1)
The initial `validate` mutation checked `task.client_nullifier !== ctx.session.nullifier` for all tasks. For agent tasks where `client_nullifier` is null, this silently passed the wrong condition. It was caught in story 3.1 spec review, but illustrates how agent/human duality creates non-obvious bugs.

### `updated_at` Forgot in `claim`
A missed `updated_at: new Date()` in the `claim` mutation is the kind of bug that's invisible in normal testing but breaks any audit trail or cache-invalidation logic. It was found during story 3.1's spec-driven review of the existing code.

### Integration Tests Deferred
The task-schema test file created in 3.1 has 9 `it.todo` integration stubs (create, claim) because no live DB is available in the test environment. These remain unimplemented and represent a significant coverage gap for the core marketplace logic.

---

## Lessons Learned

1. **Dual-client models (human vs agent) need explicit test cases for each path.** Every business rule that applies to "the client" must be separately tested for human-client and agent-client scenarios. Story 3.1's agent `validate` bug is proof.

2. **When a mutation modifies state, always include `updated_at`.** This should be a checklist item in every story that modifies DB rows, not something discovered during review.

3. **`it.todo` integration stubs are better than nothing, but they need a home.** The 9 todos in `task-schema.test.ts` should either get a test DB setup in CI or be tracked as explicit debt.
