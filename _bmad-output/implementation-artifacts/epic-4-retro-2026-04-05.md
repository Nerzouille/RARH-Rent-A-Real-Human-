## What Went Well

### Lazy Singleton Pattern for Hedera Client
Switching `getClient()` to a lazy singleton (one Hedera Client instance per server lifetime) was a clean infrastructure win. Multiple parallel requests no longer each create a new client, avoiding connection overhead and potential resource leaks.

### Transaction Memo as Lightweight Escrow
Using `escrow:taskId:budgetHbar` in Hedera TX memos as a pseudo-escrow mechanism is pragmatic and correct for the hackathon scope. It gives judges a verifiable audit trail on Hashscan without building a full multi-sig escrow contract.

### Hashscan Links Throughout the UI
Every Hedera TX ID in the UI links to Hashscan testnet. This is exactly the right judge experience — click a TX, see it on the blockchain, done. Story 4.5 completed this cohesively.

### Review Caught Security Issues Early
The `getPlatformBalance` endpoint was missing an admin role check — caught and fixed in 4.1 review. Without the code review gate, this would have been an unauthenticated endpoint exposing platform financial data.

## What Was Difficult

### Non-Atomic DB + Hedera Operations
Every payment operation (escrow lock, payment release, deposit simulation) writes to Hedera first, then updates the DB. If the DB write fails after the Hedera TX succeeds, the system ends up in an inconsistent state. This was identified and explicitly deferred in story 4.1 review. For demo it's acceptable; for production it requires an outbox/saga pattern.

### No Idempotency on Lock/Release
`lockEscrow()` and `releasePayment()` have no idempotency guards. A double-click or network retry could fire duplicate Hedera transactions. Deferred in reviews 4.1 and 4.4. This is a real risk in a live demo if a button is clicked twice.

### Env Validation at Module Load
The pattern of validating env vars at module load time (throw if missing) is good for developer experience but means the test suite can't import `hedera.ts` without valid env vars in `process.env`. Tests needed explicit environment mocking to work around this.

---

## Lessons Learned

1. **Distributed transactions need an explicit strategy defined before coding starts.** The non-atomic Hedera+DB pattern was never going to be fixable without an outbox or saga pattern. Define this upfront and decide: is it in scope or deferred? Don't discover it in review.

2. **Idempotency keys should be designed in from the start for any payment operation.** Both `lockEscrow` and `releasePayment` could have accepted an idempotency key (e.g., task ID) at no extra cost. This would have eliminated the double-fire risk entirely.

3. **Module-level env validation is a good pattern, but requires a testing strategy.** Either use dependency injection for the Hedera client or set up standard env mocking in the test suite setup.
